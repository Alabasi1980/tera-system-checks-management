## 2. ماذا نختبر

### 2.1 API Routes (Integration Tests)

| المسار | الاختبار |
|--------|---------|
| `GET /api/recipes` | يعرض الوصفات المنشورة فقط، يدعم البحث والترقيم |
| `GET /api/recipes/[slug]` | يعود بـ 404 لوصفة غير موجودة، يعرض التفاصيل كاملة |
| `POST /api/admin/recipes` | يتحقق من Admin، يرفض البيانات غير الصحيحة، ينشئ الوصفة |
| `PUT /api/admin/recipes/[id]` | يحدث البيانات، يتحقق من المكونات |
| `DELETE /api/admin/recipes/[id]` | يحذف مع Cascade |
| `POST /api/recipes/[slug]/comments` (1B) | يضيف تعليق، يتحقق من البيانات، يمنع HTML |
| `GET /api/admin/dashboard` | يعود بالبيانات الصحيحة |

### 2.2 Business Logic Tests

| القاعدة | الاختبار |
|---------|---------|
| BR-09 (اسم المكون فريد) | محاولة إضافة مكون موجود → 409 |
| BR-19 (مكون واحد على الأقل) | محاولة حفظ وصفة بدون مكونات → خطأ |
| BR-27/28 (مقياس ديناميكي) | حساب الكمية الجديدة = الأصلية × (جديد ÷ أصلي) |
| BR-03 (موافقة التعليق) | التعليق الجديد `isApproved = false` |
| Auth | الوصول إلى `/admin/*` بدون توكن → 401 |

### 2.3 UI Component Tests

| المكون | الاختبار |
|--------|---------|
| `RecipeCard` | يعرض العنوان والصورة والوقت |
| `ScalerControl` (1B) | تغيير العدد يحدث الكميات بشكل صحيح |
| `CommentSection` (1B) | إرسال تعليق، عرض التعليقات المعتمدة |
| `RecipeForm` | إضافة مكون، إضافة خطوة، رفع صورة |

كل TASK-ID يجب أن يمر بهذه الفحوصات قبل الإغلاق:

- [ ] جميع Acceptance Criteria محققة

- [ ] API Routes تعمل بشكل صحيح
- [ ] RTL صحيح
