# 09_IMPLEMENTATION_PLAN.md

## خطة التنفيذ — نظام إدارة الشيكات MVP

---

### 1. قرار التنفيذ المختصر

هذه الخطة مخصصة لتنفيذ MVP صغير لتطبيق إدارة الشيكات باستخدام القرار التقني المعتمد في `08_TECHNICAL_ARCHITECTURE.md`:

| العنصر | القرار المعتمد |
|---|---|
| التقنية | Next.js |
| اللغة | TypeScript |
| قاعدة البيانات | PostgreSQL محلية |
| ORM | Prisma |
| النمط | Full-stack Next.js بدون Backend منفصل |

لا تتضمن هذه الخطة أي كود، ولا تنشئ أي ملفات تطبيق. هي فقط ترتيب تنفيذ لاحق.

---

### 2. مبادئ التنفيذ وفق MVP Anti-Bloat

أثناء التنفيذ لاحقًا يجب الالتزام بما يلي:

1. تنفيذ الوظائف الأساسية فقط.
2. عدم إنشاء Backend منفصل.
3. عدم إنشاء طبقات Services أو Repositories إلا إذا ظهرت حاجة واضحة.
4. عدم إضافة Redux/Zustand أو إدارة حالة معقدة.
5. الالتزام بالشاشات الخمس المعتمدة فقط.
6. عدم إنشاء Status History أو Audit Log.
7. عدم إضافة Dashboard مستقلة.
8. عدم إضافة Import/Export أو إشعارات.
9. عدم تنفيذ ميزات غير موجودة في ملفات التحليل المعتمدة.

---

### 3. أول ما يجب بناؤه

أول ما يجب بناؤه لاحقًا هو **أساس المشروع وربطه بقاعدة البيانات**:

1. إنشاء مشروع Next.js باستخدام TypeScript.
2. إعداد اتصال PostgreSQL المحلي.
3. إعداد Prisma.
4. إنشاء نموذج البيانات الأساسي:
   - `users`
   - `banks`
   - `parties`
   - `checks`
5. إنشاء مستخدم Admin مبدئي للتجربة.

السبب: لا يمكن بناء الشاشات أو عمليات CRUD بشكل صحيح قبل تثبيت قاعدة البيانات والمصادقة الأساسية.

---

### 4. مراحل التنفيذ المقترحة

#### المرحلة 1 — إعداد الأساس التقني

الهدف: تجهيز المشروع للعمل محليًا فقط.

المهام:
- إنشاء مشروع Next.js + TypeScript.
- تثبيت Prisma.
- ربط PostgreSQL المحلية.
- ضبط متغير الاتصال بقاعدة البيانات محليًا.

ممنوع في هذه المرحلة:
- إعداد نشر Production.
- Docker.
- CI/CD.
- Backend منفصل.

---

#### المرحلة 2 — قاعدة البيانات ونموذج البيانات

الهدف: إنشاء الجداول الأساسية فقط.

الترتيب:
1. جدول `users`.
2. جدول `banks`.
3. جدول `parties`.
4. جدول `checks`.

ملاحظات:
- الحالات تحفظ داخل حقل `status` في جدول `checks`.
- لا يتم إنشاء جدول `status_history`.
- لا يتم إنشاء جدول `audit_logs`.
- لا يتم إنشاء جدول مستقل للحالات.

---

#### المرحلة 3 — المصادقة والصلاحيات البسيطة

الهدف: تمكين دخول المستخدمين وحماية الشاشات.

المهام:
- تسجيل دخول باسم مستخدم وكلمة مرور.
- حماية الشاشات من الوصول بدون تسجيل دخول.
- تطبيق دورين فقط:
  - `ADMIN`
  - `USER`
- منع User من إدارة البنوك والجهات والمستخدمين.

ممنوع في هذه المرحلة:
- OAuth.
- 2FA.
- تسجيل ذاتي للمستخدمين.
- صلاحيات متعددة المستويات.

---

#### المرحلة 4 — شاشة إدارة الشيكات

الهدف: بناء أهم شاشة في النظام أولًا.

المهام:
- عرض قائمة الشيكات.
- إضافة شيك.
- تعديل شيك غير نهائي.
- حذف شيك بواسطة Admin فقط.
- تغيير حالة الشيك حسب الحالات المعتمدة.
- البحث والفلترة.
- طباعة الكشف الحالي باستخدام طباعة المتصفح.

ملاحظات:
- الإضافة والتعديل والتفاصيل وتغيير الحالة تكون داخل نفس شاشة إدارة الشيكات.
- لا يتم إنشاء شاشات منفصلة للإضافة أو التعديل أو التفاصيل.

---

#### المرحلة 5 — شاشة إدارة البنوك

الهدف: إدارة البيانات المرجعية للبنوك.

المهام:
- عرض البنوك.
- إضافة بنك.
- تعديل بنك.
- حذف بنك إذا لم يكن مرتبطًا بشيكات.

الصلاحية:
- Admin فقط.

---

#### المرحلة 6 — شاشة إدارة الجهات

الهدف: إدارة الأطراف المرتبطة بالشيكات.

المهام:
- عرض الجهات.
- إضافة جهة.
- تعديل جهة.
- حذف جهة إذا لم تكن مرتبطة بشيكات.

الصلاحية:
- Admin فقط.

---

#### المرحلة 7 — شاشة إدارة المستخدمين

الهدف: إدارة مستخدمي النظام بشكل بسيط.

المهام:
- عرض المستخدمين.
- إضافة مستخدم.
- تعديل بيانات مستخدم.
- تفعيل/تعطيل مستخدم.
- تعيين الدور: Admin أو User.

الصلاحية:
- Admin فقط.

---

#### المرحلة 8 — مراجعة تشغيلية داخلية

الهدف: التأكد أن MVP يعمل كدورة كاملة.

المهام:
- تجربة تسجيل الدخول.
- إنشاء بنك وجهة.
- إنشاء شيك وارد وشيك صادر.
- تغيير الحالات.
- تجربة الفلاتر.
- تجربة الطباعة.
- التأكد من صلاحيات Admin/User.

> خطة الاختبار التفصيلية لا تُنشأ الآن، بل لاحقًا في `10_TESTING_AND_ACCEPTANCE.md` عند اعتماد الانتقال لمرحلة الاختبار.

---

### 5. ترتيب CRUD المقترح

الترتيب العملي لتقليل التعطيل أثناء التنفيذ:

1. Users/Auth.
2. Banks CRUD.
3. Parties CRUD.
4. Checks CRUD.
5. Check Status Update.
6. Filters/Search.
7. Print Statement.

السبب:
- الشيك يعتمد على البنك والجهة والمستخدم.
- لذلك يجب تجهيز البيانات المرجعية قبل إكمال شاشة الشيكات.

---

### 6. ما لا يجب تنفيذه الآن

| العنصر | سبب عدم التنفيذ الآن |
|---|---|
| Status History | تحسين مستقبلي وليس ضمن MVP |
| Audit Log | غير مطلوب الآن |
| Dashboard مستقلة | تم دمج الملخص داخل شاشة الشيكات |
| Export Excel/PDF | غير مطلوب الآن |
| Import Excel | غير مطلوب الآن |
| Notifications | غير مطلوبة الآن |
| Backend منفصل | Next.js Full-stack يكفي |
| Angular أو React standalone | القرار المعتمد Next.js |
| Docker / CI-CD | النشر مؤجل |
| تقارير متقدمة | الكشف البسيط يكفي |
| أدوار إضافية | Admin/User كافيان للنسخة الأولى |

---

### 7. الملفات أو الطبقات التي لا تُقترح الآن

لمنع التضخيم، لا يتم اقتراح إنشاء:

- Repositories.
- Services عامة مبكرًا.
- Store مركزي للحالة.
- API خارج Next.js.
- مجلدات Feature كثيرة.
- نظام صلاحيات متقدم.
- مكونات UI كثيرة قبل الحاجة.

يمكن إضافة مكون أو ملف فقط إذا كان يخدم شاشة أو وظيفة معتمدة مباشرة.

---

### 8. متى يمكن توليد EngineeringAgent لاحقًا؟

يمكن توليد `EngineeringAgent` فقط بعد تحقق الشروط التالية:

1. اعتماد هذا الملف `09_IMPLEMENTATION_PLAN.md` من صاحب المشروع.
2. تحديد أن التنفيذ البرمجي سيبدأ فعليًا.
3. تحديد أول مهمة تنفيذية صغيرة بوضوح.
4. تحديد الملفات المسموح بإنشائها أو تعديلها.
5. تأكيد عدم وجود تعديل إضافي على التحليل أو المعمارية.

أول مهمة مناسبة لـ EngineeringAgent لاحقًا قد تكون:

```text
إنشاء مشروع Next.js + TypeScript وربطه مبدئيًا بـ PostgreSQL وPrisma، بدون بناء الشاشات بعد.
```

لكن لا يتم ذلك الآن.

---

### 9. قرار تيرا التنفيذي

```text
قرار تيرا:
- خطة التنفيذ مناسبة لـ MVP صغير.
- يبدأ التنفيذ لاحقًا من الأساس التقني وقاعدة البيانات.
- ثم المصادقة.
- ثم شاشات CRUD حسب الترتيب المعتمد.
- لا يتم تنفيذ ميزات مستقبلية الآن.
- لا يتم توليد EngineeringAgent إلا بعد اعتماد هذه الخطة وطلب بدء التنفيذ.

الخطوة التالية:
- انتظار موافقة صاحب المشروع على خطة التنفيذ قبل توليد EngineeringAgent أو إنشاء أي كود.
```
