# 10_TESTING_AND_ACCEPTANCE.md

## اختبار وقبول نظام إدارة الشيكات — MVP

---

### 1. هدف هذا الملف

هذا الملف يحدد اختبارات ومعايير قبول النسخة الأولى MVP من نظام إدارة الشيكات.

لا يحتوي هذا الملف على كود، ولا يطلب إنشاء ملفات تطبيق. الهدف هو تجهيز معايير قبول واضحة قبل التنفيذ البرمجي.

---

### 2. نطاق الاختبار في MVP

الاختبار الحالي يغطي فقط الوظائف المعتمدة في ملفات التحليل والخطة:

- تسجيل الدخول.
- صلاحيات Admin وUser.
- إدارة البنوك.
- إدارة الجهات.
- إدارة الشيكات الصادرة والواردة.
- تغيير حالات الشيك.
- البحث والفلترة.
- الطباعة البسيطة.
- منع أخطاء الإدخال الأساسية.

---

### 3. معايير قبول MVP العامة

يُعتبر MVP مقبولًا مبدئيًا إذا تحقق ما يلي:

| المعيار | شرط القبول |
|---|---|
| تسجيل الدخول | لا يمكن الدخول للنظام بدون بيانات صحيحة |
| الصلاحيات | Admin يرى ويدير كل الأقسام، User يعمل على الشيكات فقط |
| البنوك | يمكن لـ Admin إضافة/تعديل/حذف بنك غير مرتبط بشيكات |
| الجهات | يمكن لـ Admin إضافة/تعديل/حذف جهة غير مرتبطة بشيكات |
| الشيكات | يمكن إنشاء شيك صادر ووارد ببيانات صحيحة |
| الحالات | يمكن تغيير حالة الشيك وفق الانتقالات المسموحة فقط |
| الفلاتر | يمكن البحث والفلترة حسب النوع، الحالة، البنك، الجهة، والتاريخ |
| الطباعة | يمكن طباعة الكشف الحالي بعد تطبيق الفلاتر |
| منع الأخطاء | النظام يمنع البيانات الناقصة أو غير الصحيحة الأساسية |
| Anti-Bloat | لا توجد ميزات خارج MVP مثل Audit Log أو Export/Import أو Dashboard مستقلة |

### 13. ما هو خارج الاختبار الآن؟

| العنصر | سبب عدم اختباره الآن |
|---|---|
| Audit Log | غير موجود في MVP |
| Status History | غير موجود في MVP |
| Notifications | غير مطلوبة في MVP |
| Export/Import | غير مطلوب في MVP |
| Dashboard مستقلة | غير موجودة، الملخص مدمج في شاشة الشيكات |
| Deployment | لم تبدأ مرحلة النشر |
| CI/CD | غير مطلوب الآن |
| Performance Testing | لا توجد متطلبات أداء خاصة الآن |
| Security Testing متقدم | الصلاحيات بسيطة ولا توجد مراجعة أمنية متخصصة الآن |

### 14. تطبيق قواعد MVP Anti-Bloat في الاختبار

لمنع تضخيم الاختبارات في هذه المرحلة:

- لا يتم اختبار ميزات غير موجودة في MVP.
- لا يتم طلب بناء تقارير متقدمة من أجل الاختبار.
- لا يتم طلب Audit Log أو Status History.
- لا يتم إضافة سيناريوهات تكامل خارجي.
- لا يتم اختبار Export/Import.
- يتم التركيز فقط على دورة العمل الأساسية من تسجيل الدخول حتى تسجيل الشيك وتغيير حالته وطباعة كشفه.

### 15. قرار قبول MVP

```text
يُقبل MVP مبدئيًا إذا:
- نجح تسجيل الدخول.
- نجحت صلاحيات Admin/User.
- نجحت إدارة البنوك والجهات بواسطة Admin.
- نجح إنشاء وتعديل الشيكات الصادرة والواردة.
- نجحت حالات الشيك الأساسية وانتقالاتها.
- نجحت الفلاتر والبحث.
- نجحت الطباعة البسيطة.
- تم منع أخطاء الإدخال الأساسية.
- لم تظهر ميزات خارج النطاق داخل التطبيق.
```

### 16. الخطوة التالية

بعد اعتماد هذا الملف، يمكن اعتبار المشروع جاهزًا مبدئيًا للانتقال إلى مرحلة التنفيذ، لكن التنفيذ لا يبدأ إلا بعد:

1. موافقة صاحب المشروع الصريحة على بدء البرمجة.
2. توليد `EngineeringAgent` عند الحاجة.
3. تحديد أول مهمة تنفيذية صغيرة ومحدودة.
