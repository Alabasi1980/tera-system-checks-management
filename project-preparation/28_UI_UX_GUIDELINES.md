# 28_UI_UX_GUIDELINES.md

## دليل تصميم واجهة المستخدم — نظام إدارة الشيكات MVP

---

### 1. Design Source Mode

| البند | القرار |
|---|---|
| Design Source Mode | Tera-Decided Design |
| نوع التصميم | Clean Administrative UI |
| الثيم | Light Theme فقط |
| اتجاه الواجهة | RTL أساسًا للواجهة العربية |
| مصدر خارجي | لا يوجد |
| getdesign.md | غير مستخدم الآن |
| CSS خارجي | غير مستخدم الآن |

---

### 2. الهدف البصري للتطبيق

الهدف هو واجهة إدارية نظيفة وبسيطة تساعد المستخدم على:

- تسجيل الشيكات بسرعة.
- قراءة الجداول بوضوح.
- تمييز حالات الشيك بدون مؤثرات مبالغ فيها.
- استخدام النماذج دون ازدحام بصري.
- تنفيذ العمليات الأساسية بأقل تشتيت.

التصميم يجب أن يكون عمليًا وواضحًا أكثر من كونه استعراضيًا.

---

### 3. الألوان العامة

يجب استخدام ألوان حيادية وهادئة مناسبة لتطبيق إداري.

| الاستخدام | اللون المقترح | ملاحظات |
|---|---|---|
| خلفية الصفحة | أبيض أو رمادي فاتح جدًا | مثل `#FFFFFF` أو `#F8FAFC` |
| خلفية البطاقات والجداول | أبيض | مع حدود خفيفة |
| النص الأساسي | رمادي داكن | مثل `#111827` |
| النص الثانوي | رمادي متوسط | مثل `#6B7280` |
| الحدود | رمادي فاتح | مثل `#E5E7EB` |
| اللون الأساسي | أزرق إداري هادئ | مثل `#2563EB` |
| نجاح | أخضر هادئ | للحالات الناجحة فقط |
| تحذير | أصفر/برتقالي هادئ | للحالات التي تحتاج انتباه |
| خطأ/منع | أحمر هادئ | للأخطاء أو العمليات الممنوعة |

لا يجوز إضافة ألوان جديدة إلا بقرار من Tera.

---

### 4. Typography مبدئي

- استخدام خط واضح يدعم العربية جيدًا.
- يفضل استخدام خط نظام آمن في البداية بدون تحميل خطوط خارجية.
- حجم النص الأساسي: متوسط ومقروء.
- العناوين واضحة لكن غير ضخمة.
- لا تستخدم خطوط زخرفية أو غير إدارية.

اقتراح مبدئي:

```text
font-family: system-ui, Arial, sans-serif
```

---

### 5. قواعد الجداول

الجداول هي أهم عنصر في التطبيق، خاصة شاشة إدارة الشيكات.

قواعد الجداول:

- صفوف واضحة بارتفاع مريح.
- حدود خفيفة بين الصفوف.
- تمييز بسيط عند المرور على الصف Hover.
- رؤوس أعمدة واضحة بخلفية رمادية فاتحة.
- محاذاة الأرقام والمبالغ بشكل واضح.
- عرض حالة الشيك Badge بسيط بلون هادئ.
- عدم استخدام ألوان قوية أو تأثيرات كثيرة.
- يجب أن تبقى أزرار الإجراءات واضحة ومحدودة.

---

### 6. قواعد النماذج

قواعد النماذج:

- الحقول مرتبة حسب منطق الإدخال.
- كل حقل له Label واضح.
- الحقول المطلوبة يجب أن تكون واضحة.
- رسائل الخطأ تظهر أسفل الحقل أو بالقرب منه.
- لا تستخدم Placeholder بدل Label.
- تجنب النماذج الطويلة جدًا.
- الإضافة والتعديل يستخدمان نفس النموذج عند الإمكان.

---

### 7. قواعد الأزرار

| نوع الزر | الاستخدام |
|---|---|
| Primary | الحفظ، إضافة شيك، تأكيد العملية الأساسية |
| Secondary | إلغاء، رجوع، عرض تفاصيل |
| Danger | حذف فقط |
| Ghost/Link | إجراءات ثانوية داخل الجدول |

قواعد عامة:

- لا تستخدم أكثر من زر Primary واحد في نفس السياق.
- زر الحذف يجب أن يكون واضحًا لكن غير مبالغ فيه.
- العمليات الخطرة تحتاج تأكيد.

---

### 8. قواعد Spacing/Layout

- استخدام مسافات ثابتة ومريحة بين العناصر.
- الواجهة تعتمد على تخطيط إداري بسيط:
  - شريط تنقل علوي أو جانبي.
  - منطقة محتوى رئيسية.
  - جداول ونماذج داخل Cards بسيطة.
- لا تستخدم Layout معقدًا أو مزدحمًا.
- يجب أن تكون شاشة إدارة الشيكات هي الشاشة الرئيسية.
- لا توجد Dashboard مستقلة في MVP.

---

### 9. ما هو ممنوع بصريًا

ممنوع في هذا التطبيق:

- ألوان عشوائية غير موثقة.
- Gradients قوية.
- مؤثرات Animation مبالغ فيها.
- ظلال ثقيلة.
- مكونات UI غير متناسقة.
- ثيم داكن Dark Mode.
- استخدام CSS خارجي غير معتمد.
- استخدام getdesign.md أو تصميم خارجي غير مذكور.
- إضافة Dashboard استعراضية أو رسوم بيانية غير مطلوبة.
- استخدام مكتبة UI تغير شكل التطبيق جذريًا دون موافقة Tera.

---

### 10. التزام EngineeringAgent لاحقًا

عند توليد EngineeringAgent لاحقًا أو بدء التنفيذ، يجب أن يلتزم بما يلي:

1. قراءة هذا الملف قبل تنفيذ أي واجهة.
2. عدم اختراع ألوان أو components خارج هذا الدليل.
3. الالتزام بـ Light Theme فقط.
4. الالتزام بواجهة RTL عربية.
5. بناء واجهة إدارية بسيطة وواضحة.
6. عدم إنشاء Dashboard مستقلة.
7. عدم إضافة مؤثرات أو مكتبات UI غير معتمدة.
8. استخدام مكونات قليلة ومباشرة تخدم الشاشات الخمس فقط.
9. أي تغيير بصري جوهري يحتاج موافقة Tera قبل التنفيذ.

---

### 11. قرار تيرا للتصميم

```text
قرار تيرا:
- مصدر التصميم: Tera-Decided Design.
- النمط: Clean Administrative UI.
- الثيم: Light Theme فقط.
- لا يوجد CSS خارجي.
- لا يوجد getdesign.md.
- EngineeringAgent لا يحق له اختراع ألوان أو مكونات خارج هذا الدليل.

الخطوة التالية:
- بعد اعتماد هذا الدليل، يمكن الانتقال لاحقًا إلى توليد EngineeringAgent عند بدء التنفيذ البرمجي.
```
