# 08_TECHNICAL_ARCHITECTURE.md

## المعمارية التقنية المقترحة — نظام إدارة الشيكات MVP

---

### 1. قرار المعمارية بعد تحديث المعطيات

بناءً على رغبة صاحب المشروع في استخدام إحدى التقنيات التالية:

- Next.js
- React
- Angular

ومع توفر قاعدة بيانات **PostgreSQL محلية** على الجهاز، فإن الاختيار الأنسب لهذا المشروع هو:

| العنصر | القرار المقترح |
|---|---|
| نوع التطبيق | Web App |
| التقنية المقترحة | **Next.js** |
| اللغة | TypeScript |
| قاعدة البيانات | PostgreSQL محلية |
| ORM المقترح | Prisma |
| نمط التطبيق | Full-stack Next.js بسيط |
| الواجهة | React داخل Next.js |
| API | Route Handlers داخل Next.js عند الحاجة فقط |
| المصادقة | تسجيل دخول بسيط محلي |

---

### 2. لماذا Next.js هو الأفضل هنا؟

Next.js هو الاختيار الأنسب بين Next.js / React / Angular لهذا المشروع للأسباب التالية:

- يوفر Frontend وBackend خفيفين داخل مشروع واحد.
- مناسب لتطبيقات إدارية صغيرة مثل إدارة الشيكات.
- يقلل الحاجة لإنشاء Backend منفصل.
- يدعم React مباشرة، لذلك يعطي مرونة React بدون تعقيد مشروع React مستقل مع API منفصل.
- أبسط من Angular لهذا الحجم.
- مناسب للتعامل مع PostgreSQL عبر Prisma.
- يسمح ببناء شاشات CRUD بسرعة وبعدد ملفات محدود.

---

### 3. لماذا لم يتم اختيار React فقط؟

React وحده مكتبة واجهة فقط، وسيحتاج غالبًا إلى:

- Backend منفصل.
- API منفصل.
- إعداد Routing وData Fetching بشكل يدوي أكثر.
- هيكلة إضافية للتعامل مع قاعدة البيانات.

لذلك React وحده سيزيد التعقيد مقارنة بـ Next.js، بينما Next.js يستخدم React بالفعل ويضيف له البنية اللازمة لتطبيق Full-stack بسيط.

---

### 4. لماذا لم يتم اختيار Angular؟

Angular قوي ومنظم، لكنه أكبر من حاجة هذا المشروع لأن:

- يتطلب بنية أثقل.
- مناسب أكثر للأنظمة الكبيرة أو فرق التطوير الكبيرة.
- يحتاج غالبًا إلى Backend منفصل.
- سيزيد عدد الملفات والمفاهيم مقارنة بحجم MVP الحالي.

لذلك Angular غير مفضل هنا لتجنب التضخيم.

---

### 5. قاعدة البيانات المقترحة

| العنصر | القرار |
|---|---|
| قاعدة البيانات | PostgreSQL محلية |
| السبب | متوفرة على الجهاز، قوية، مناسبة للبيانات المالية والإدارية، وقابلة للتوسع لاحقًا |
| طريقة الاتصال | عبر Prisma ORM |
| بيئة التشغيل الحالية | محلية أثناء التطوير |

قاعدة PostgreSQL مناسبة لهذا التطبيق لأنها تحفظ بيانات مترابطة:

- الشيكات.
- البنوك.
- الجهات.
- المستخدمين.

---

### 6. ORM المقترح: Prisma

Prisma مناسب لهذا المشروع لأنه:

- يعمل جيدًا مع PostgreSQL.
- يقلل أخطاء كتابة SQL اليدوي.
- يوفر نموذج بيانات واضح.
- مناسب مع Next.js وTypeScript.
- يساعد لاحقًا في إنشاء Migrations بطريقة منظمة.

> هذا لا يعني إنشاء كود الآن. هو قرار معماري فقط.

---

### 7. بنية المشروع المقترحة لاحقًا — بدون إنشاء كود الآن

هذه بنية مبدئية فقط لتوجيه التنفيذ لاحقًا، ولا تعني إنشاء الملفات الآن:

```text
app/
  login/
  checks/
  banks/
  parties/
  users/

app/api/
  auth/
  checks/
  banks/
  parties/
  users/

components/
  CheckForm
  CheckTable
  StatusBadge
  ConfirmDialog

lib/
  prisma
  auth
  permissions

prisma/
  schema.prisma
```

---

### 8. تقليل الطبقات ومنع التضخيم

لأن المشروع MVP صغير، لا يتم إنشاء طبقات كثيرة في البداية.

| الطبقة/العنصر | القرار |
|---|---|
| Pages / App Routes | مطلوبة للشاشات الخمس فقط |
| API Routes | تستخدم فقط عند الحاجة للعمليات CRUD |
| Prisma Client | مطلوب للتعامل مع PostgreSQL |
| Components | تستخدم للمكونات المتكررة فقط |
| Services | لا تُنشأ إلا إذا تكرر المنطق بوضوح |
| Repositories | غير مطلوبة الآن |
| State Management مثل Redux | غير مطلوب الآن |
| UI Framework كبير | غير مطلوب إلا إذا قرر المستخدم لاحقًا |
| API منفصل خارج Next.js | غير مطلوب الآن |

---

### 9. الشاشات المعتمدة في التنفيذ لاحقًا

يجب الالتزام بالشاشات الخمس المعتمدة في `07_SCREENS_AND_UI_STRUCTURE.md`:

1. تسجيل الدخول.
2. إدارة الشيكات.
3. إدارة البنوك.
4. إدارة الجهات.
5. إدارة المستخدمين.

لا يتم إنشاء شاشات إضافة/تعديل منفصلة إلا إذا ظهرت حاجة فعلية لاحقًا.

---

### 10. نموذج البيانات التقني المعتمد

يعتمد التنفيذ لاحقًا على نموذج البيانات المبسط في `06_DATA_MODEL_PREPARATION.md`:

- `checks`
- `banks`
- `parties`
- `users`

حالات الشيك تحفظ داخل حقل `status` في جدول `checks`.

---

### 11. Status History / Audit Log

وفق قرار MVP Anti-Bloat:

- لا يتم إنشاء `Status History` الآن.
- لا يتم إنشاء `Audit Log` الآن.
- لا يتم إنشاء جدول مستقل للحالات الآن.

لكن يمكن اعتبارها تحسينات مستقبلية إذا ظهرت حاجة إلى:

- معرفة من غيّر حالة الشيك ومتى.
- تتبع التعديلات والحذف.
- متطلبات تدقيق داخلية أو قانونية.
- مسؤولية تشغيلية أعلى.

في النسخة الحالية، يكفي حفظ:

- الحالة الحالية `status`.
- آخر ملاحظة حالة `status_note`.
- تاريخ آخر تغيير حالة `status_changed_at`.

---

### 12. المصادقة والصلاحيات

المطلوب في MVP:

- تسجيل دخول باسم مستخدم وكلمة مرور.
- دوران فقط:
  - `ADMIN`
  - `USER`
- Admin يدير الشيكات والبنوك والجهات والمستخدمين.
- User يدير الشيكات فقط.

غير مطلوب الآن:

- OAuth.
- 2FA.
- صلاحيات متعددة المستويات.
- تسجيل ذاتي للمستخدمين.

---

### 13. ما هو خارج المعمارية الآن؟

| العنصر | سبب التأجيل |
|---|---|
| Backend منفصل | Next.js يكفي كتطبيق Full-stack بسيط |
| React standalone | يحتاج Backend منفصل غالبًا |
| Angular | أكبر من حاجة MVP الحالي |
| Redux أو Zustand | لا حاجة لإدارة حالة معقدة |
| Status History كامل | تحسين مستقبلي وليس ضمن MVP |
| Audit Log كامل | تحسين مستقبلي وليس ضمن MVP |
| Notifications | غير مطلوبة الآن |
| Import/Export Excel | غير مطلوب الآن |
| Deployment Plan | لم تبدأ مرحلة النشر |
| Docker / CI/CD | غير مطلوب الآن |
| Testing Plan تفصيلي | مؤجل إلى مرحلة لاحقة |
| Implementation Plan | مؤجل إلى مرحلة لاحقة |

---

### 14. قواعد Anti-Bloat أثناء التنفيذ لاحقًا

عند بدء التنفيذ لاحقًا يجب الالتزام بما يلي:

1. إنشاء أقل عدد ملفات ممكن لتنفيذ الوظيفة المطلوبة.
2. عدم إنشاء Services أو Repositories إلا إذا تكرر المنطق بوضوح.
3. عدم فصل Frontend وBackend.
4. عدم إنشاء Backend مستقل خارج Next.js.
5. بناء 5 شاشات فقط كما هو محدد في ملف الشاشات.
6. عدم إضافة Dashboard مستقلة.
7. عدم إنشاء Status History أو Audit Log في MVP.
8. عدم إضافة تقارير متقدمة أو تصدير Excel بدون اعتماد جديد.
9. عدم إضافة State Management معقد.
10. أي توسعة يجب أن تمر عبر قرار تيرا وموافقة المستخدم.

---

### 15. قرار تيرا التقني المحدث

```text
قرار تيرا:
- المعمارية المقترحة: Next.js Full-stack Web App.
- اللغة المقترحة: TypeScript.
- قاعدة البيانات: PostgreSQL محلية.
- ORM: Prisma.
- لا يوجد Backend منفصل الآن.
- لا يوجد Angular الآن.
- لا يوجد React standalone الآن.
- لا يوجد Status History أو Audit Log في MVP.
- لا يوجد تنفيذ برمجي في هذه المرحلة.

سبب القرار:
- Next.js يحقق توازنًا مناسبًا بين React وBackend بسيط داخل مشروع واحد.
- PostgreSQL متوفرة محليًا ومناسبة للبيانات الإدارية والمالية.
- Prisma يقلل التعقيد في التعامل مع قاعدة البيانات.
- المعمارية مناسبة لمشروع MVP صغير وقابلة للتوسع لاحقًا.

الخطوة التالية:
- انتظار موافقة صاحب المشروع قبل إنشاء خطة التنفيذ أو توليد EngineeringAgent.
```
