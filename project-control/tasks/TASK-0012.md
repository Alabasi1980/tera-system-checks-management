## 5. Key Design Decisions

1. **Auth gating**: `requireAdmin()` only on `deleteCheck`. All other CRUD + status actions use `getSession()` (auth required, not admin).
2. **Bank/Party dropdowns**: New lightweight actions `getBanksForDropdown()` and `getPartiesForDropdown()` using `getSession()` — NOT the existing Admin-gated `listBanks`/`listParties`.
3. **Status transitions**: Enforced both server-side (validation matrix) and client-side (only valid options shown).
4. **Print**: `window.print()` with `@media print` styles in component.
5. **Filters**: Client-side or URL search params.
6. **UI pattern**: Identical to Banks/Parties — inline `styles` object, `'use client'`, modals, toast.

## 7. Sub-Task 1: Backend — Checks Server Actions

### Agent
EngineeringAgent

### Objective
Create `app/checks/actions.ts` with all server-side actions for the Checks screen. All actions require auth (`getSession()`) except `deleteCheck` which requires Admin (`requireAdmin()`).

### Allowed Write Targets
- `checks-management/app/checks/actions.ts` — **CREATE** (new file)

### Reference Files
- `checks-management/app/banks/actions.ts` — pattern: validation, normalization, `trimValue`, error returns
- `checks-management/app/parties/actions.ts` — same pattern
- `checks-management/prisma/schema.prisma` — Check model, enums (CheckType, CheckStatus), relations
- `checks-management/lib/auth.ts` — `getSession()` and `requireAdmin()`
- `checks-management/lib/prisma.ts` — Prisma client
- `project-control/workflow-rules.md` — transition matrix, badge colors
- `project-preparation/06_DATA_MODEL_PREPARATION.md` — field types and constraints
- `project-preparation/07_SCREENS_AND_UI_STRUCTURE.md` — S02 spec

### Required Server Actions

1. **`listChecks(filters?)`** — List with optional filters:
   - Filters: `type` (CheckType), `status` (CheckStatus), `bankId` (number), `partyId` (number), `dueDateFrom`/`dueDateTo` (Date), `search` (string searches checkNumber or party.name)
   - Returns joined data: bank name, party name, creator username
   - Ordered by `createdAt DESC`
   - Uses `getSession()` — requires auth, not Admin

2. **`getCheckSummary()`** — Summary statistics:
   - Returns: `totalCount`, `totalAmount`, `dueSoonCount` (due within 7 days, not final), `returnedCount` (status RETURNED)
   - Uses `getSession()` — requires auth, not Admin

3. **`getBanksForDropdown()`** — Banks for dropdown:
   - Returns `{ id: number; name: string }[]` ordered by name ASC
   - Uses `getSession()` — NOT requireAdmin

4. **`getPartiesForDropdown()`** — Parties for dropdown:
   - Same pattern as banks
   - Uses `getSession()` — NOT requireAdmin

5. **`createCheck(data)`** — Create check:
   - Fields: `type`, `bankId`, `partyId`, `checkNumber`, `amount`, `issueDate`, `dueDate`, `notes?`
   - Validation: required fields, amount > 0, dueDate >= issueDate, maxLengths
   - Verify bankId/partyId exist
   - Set status = REGISTERED, createdById from session
   - Uses `getSession()` — requires auth, not Admin

6. **`updateCheck(id, data)`** — Update check:
   - Same validation as create
   - BLOCK if check is CASHED or CANCELLED (final state)
   - Uses `getSession()` — requires auth, not Admin

7. **`updateCheckStatus(id, newStatus, statusNote?)`** — Status transition:
   - Validate transition against matrix:
     - REGISTERED → IN_HAND, CANCELLED
     - IN_HAND → CASHED, RETURNED, CANCELLED
     - CASHED → (none — final)
     - RETURNED → IN_HAND, CASHED, CANCELLED
     - CANCELLED → (none — final)
   - Set statusChangedAt = new Date()
   - Store statusNote if provided
   - BLOCK if check is in final state
   - Uses `getSession()` — requires auth, not Admin

8. **`deleteCheck(id)`** — Delete check:
   - Call `requireAdmin()` — checks auth AND Admin role
   - BLOCK if check is CASHED or CANCELLED
   - Uses `requireAdmin()` — Admin only

9. **`getCheckById(id)`** — Single check full details:
   - Returns all fields + joined names
   - Uses `getSession()` — requires auth, not Admin

### Exported Types
- `CheckItem`, `CheckDetail`, `CheckFormData`, `CheckSummary`, `DropdownItem`

### Security Notes
- `deleteCheck` is the ONLY Admin-gated action
- All others use `getSession()` (auth required only)
- Server-side validation must mirror client-side: amount > 0, dueDate >= issueDate
- Must verify bankId/partyId exist in DB before creating/updating
- Amount must be parsed as Decimal, returned as number

### Forbidden Actions
- Do NOT modify `middleware.ts`, `lib/auth.ts`, `lib/prisma.ts`, `schema.prisma`
- Do NOT modify Banks or Parties actions
- Do NOT create migrations or UI files
- Do NOT add status history or audit log
- Do NOT add tests

### Acceptance Criteria
1. `listChecks` returns items with bank/party/creator names, supports all filters, sorted by createdAt DESC
2. `getCheckSummary` returns accurate counts and amount
3. `getBanksForDropdown` / `getPartiesForDropdown` work without Admin role
4. `createCheck` saves with status=REGISTERED, validates amount>0 and dueDate>=issueDate
5. `updateCheck` blocks if check is in final state (CASHED/CANCELLED)
6. `updateCheckStatus` validates transition matrix, blocks invalid/final transitions
7. `deleteCheck` requires Admin, blocks deletion of final-state checks
8. `getCheckById` returns full details with joined names
9. All actions return `{ success: true }` or `{ error: string }`
10. No real secrets in code, no schema changes, no migrations
