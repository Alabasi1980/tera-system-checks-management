## Objective

الهدف من هذه المهمة هو مراجعة أمنية مستقلة وشاملة لمخرجات TASK-0004 (Authentication + Roles).

**لماذا الآن؟**
- TASK-0004 أنشأت نظام Auth متكامل (JWT، Cookies، Middleware، كلمات مرور، صلاحيات)
- حدث تسرب سابق في ISSUE-0003 (Secret Exposure)
- DEC-0006 ينص صراحة على أن SecurityAgent يجب أن يراجع قبل الانتقال للمرحلة التالية
- المهمة التالية (Phase 4 — شاشة إدارة الشيكات) ستعتمد على Auth

### المراجعة تغطي:

1. **JWT Implementation**: إنشاء التوكن، التحقق منه، استخدامه في Middleware
2. **Cookie Security**: httpOnly, secure, sameSite, maxAge, مسح الـ Cookie
3. **Password Handling**: تشفير bcryptjs، عدم تخزين plaintext، استخدام env vars
4. **Middleware Logic**: حماية Routes، عدم وجود bypass، فحص الصلاحيات
5. **Role Enforcement**: فصل Admin/User، منع وصول User لمسارات Admin
6. **Secret Isolation**: عدم وجود hardcoded secrets في الكود أو config
7. **Seed Script**: عدم وجود كلمة مرور حقيقية في ملفات الكود
8. **User Enumeration Prevention**: رسائل خطأ غير كاشفة
9. **Session Management**: مدة الصلاحية، إمكانية إلغاء الجلسة

## Focus Areas (Security Checklist)

| # | المجال | الوصف |
|---|---|---|
| 1 | JWT Secret | هل `JWT_SECRET` قوي وعشوائي؟ هل هو في `.env` فقط؟ |
| 2 | Token Creation | هل `createToken` تستخدم الخوارزمية الصحيحة (`HS256`/`HS384`)? هل الـ payload مناسب؟ |
| 3 | Token Verification | هل `verifyToken` تتعامل مع token منتهي الصلاحية أو غير صالح؟ |
| 4 | Cookie Settings | هل الـ Cookie: `httpOnly`, `secure` (في production), `sameSite`, `maxAge`؟ |
| 5 | Logout | هل مسح الـ Cookie صحيح وآمن؟ |
| 6 | Middleware Protection | هل جميع Routes محمية ما عدا `/login` و `/_next/`؟ |
| 7 | Role Bypass | هل يمكن لمستخدم User الوصول لـ `/users`, `/banks`, `/parties`؟ |
| 8 | Password Storage | هل `hashPassword` تستخدم bcrypt؟ هل salt rounds كافية؟ |
| 9 | Error Messages | هل رسائل الخطأ تكشف إن كان المستخدم موجودًا؟ (User Enumeration) |
